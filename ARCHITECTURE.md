# Architecture

RedAnts is an Umbraco 17 / .NET 10 application combining a public marketing website with a self-service ticketing system, backed by SQLite.

## Layering (ports and adapters)

Three top-level layers, with dependencies pointing inward (`Infrastructure` → `Features` → `Domain`):

### `Domain/`
Pure business model: entities, enums, value objects, and `DomainException`. No dependency on Umbraco, ASP.NET, or any framework. Examples: `Domain/Ticketing/` (`Event`, `Season`, `Venue`, `BillingAddress`, `PostalCode`) and `Domain/Ticketing/Sales/` (`Order`, `EventTicket`, `SeasonSingleTicket`, `SeasonPass`, `MemberCard`, `EventVisit`/`EventVisitLog`/`EventFreeEntry`, the pricing aggregates `EventPrice`/`SeasonPrice`/`CategoryPrice`, and the enums in `SalesEnums.cs`).

### `Features/`
Application layer, organized by use case rather than by technical type:

- `Features/Ticketing/Ports/` defines the interfaces the application depends on: `CatalogPorts` (read-only Season/Venue/Event) and `SalesCatalogPorts` (pricing read/write `IEventPrices`/`ISeasonPrices`, the resolved read side `IEventPricing`, issued `IEventTickets`).
- Use-case folders hold controllers and Blazor components: `Public/` (browsing, access gate), `Cart/` (guest shopping cart), `Admin/` (backoffice ticketing section and its admission-count report models), `Scanning/` (QR scanner).

Controllers depend only on ports, never on concrete infrastructure.

### `Infrastructure/`
Adapters that implement the ports and integrate with Umbraco and external services. Organized by slice:

- `Infrastructure/Shared/`: cross-cutting helpers (`SqidEncoder`, `EmailLayout`).
- `Infrastructure/Ticketing/`: catalog readers over Umbraco content (`UmbracoCatalogReaders`), the `Sales/` slice (NPoco records + repositories for orders/tickets/pricing, `PricingCatalogSeeder`), the single-step schema `TicketingMigration`, `SessionCartService`, the `Admin/` report readers, and the `TicketingComposer` that wires it up.
- `Infrastructure/Website/`: the public website content types and seeding.

## Two content slices

Two independent slices share the Umbraco instance and must stay decoupled:

1. **Ticketing**: events, seasons, event tickets / season single tickets / season passes / member cards, a guest shopping cart, per-event and per-season pricing with quotas, admission scanning, and a purchase flow with Payrexx payment and Brevo confirmation email.
2. **Website**: the public marketing site. A `flexPage` document type holds a `content` Block List of element types (`heroElement`, `headerElement`, `eventListElement`), rendered through `Views/FlexPage.cshtml`, `Views/Shared/_SiteLayout.cshtml`, and `Views/Partials/Blocks/{alias}.cshtml`.

Website code must not reach into ticketing internals. If a real dependency arises, route it through a port. The event-list block is the sanctioned bridge: it reads ticketing seasons through the published-content API, not through ticketing internals.

## Ticketing data model

The catalog (Season, Venue, Event) is Umbraco Document Types. Everything transactional is NPoco tables, built by `CreateTicketingSchema` in `Infrastructure/Ticketing/TicketingMigration.cs`. The build is fully idempotent and re-runs on every boot: `TicketingMigrationComponent` resets the recorded migration state before each run, and every table is created only when missing (`EnsureTable`/`TableExists`) while additive columns are gated by `ColumnExists`. A new table or column therefore appears on the next start with no database drop, and the "recorded state says applied but the table is gone" desync (which happened when a dev database was swapped between parallel sessions) cannot occur. See `README.md` for the full entity-relationship diagram and enum value tables.

Design rules:

- **Enums are persisted as their integer value.** `Category` (`TicketCategory`), `Status` (`OrderStatus`/`TicketStatus`), `PaymentMethod`, `TicketType`, `FreeEntryType`, and the visit-log `Type` (`VisitLogType`) are all `int` columns. Records store `(int)theEnum`; repositories cast back, and raw SQL in the admin report readers compares against `(int)` casts. There is no category master table and no `CategoryCode`/`CategoryName` — the display label comes from `TicketCategory.DisplayName()`.
- **No enforced foreign keys.** Tables reference each other (and the Umbraco catalog) by id only; `EventId`/`SeasonId` hold the Umbraco content node id. This keeps the ticketing tables decoupled from the content tree and from each other.
- **Pricing (`EventPrices`/`SeasonPrices`) is 0..1 per event/season** (unique on the node id), each with n category sub-rows (`Category`, `SalePrice`, `Quota?`). `EventPrices` additionally carries the event-level `TotalSalesQuota` (Verkaufskontingent gesamt) and `AdmissionQuota` (Einlasskontingent, persons). `EventPricingReader` marks a category sold out once its `Quota` or the event's `TotalSalesQuota` is reached by the valid issued `EventTickets`.
- **Admissions are one `TicketEventVisits` row per `(event, ticket)`.** There is no `CheckedOutAt`; the individual in/out scans are appended to `TicketEventVisitsLogs`, and `IsInside` is the visit's current presence. Free-entry persons (players, staff, officials, Swiss Unihockey free cards) are admitted spontaneously with `TicketType = FreeEntry` (`TicketUuid` null) and their kind recorded in `TicketEventFreeEntries` (`FreeEntryType`).
- `EventTicket` and `SeasonSingleTicket` carry a `bool Redeemed` (the single-admission tickets); `SeasonPass` and `MemberCard` are multi-event and have no redemption flag. `Orders` is the immutable financial record; tickets link to it by optional `OrderId`.

`PricingCatalogSeeder` idempotently gives every event and season a base price set (Adult/Youth/Child) on boot and seeds a few demo `EventTickets`.

## Code-first content types

Content types are defined and maintained in code, not by hand in the backoffice. Each slice follows the same pattern:

1. A `IComposer` registers an `INotificationAsyncHandler<UmbracoApplicationStartedNotification>`.
2. On application start the handler ensures document types, element types, and data types (e.g. the Block List) exist, then seeds sample content.
3. Every step is **idempotent** (check-then-create / reconcile), so it runs safely on every boot and on an existing database.
4. All type and property aliases live in one `*Aliases` static class per slice (`TicketingAliases`, `WebsiteAliases`). Never scatter alias string literals through the code.

Reference implementations: `Infrastructure/Ticketing/Content/TicketingContentTypeSeeder.cs`, `Infrastructure/Website/WebsiteContentTypeSeeder.cs`.

### Block List value format (Umbraco v17)

Block List content values are JSON with `contentData` (array of `{contentTypeKey, key, values:[{alias,culture,segment,value}]}`), optional `settingsData`, an `expose` array (`{contentKey,culture,segment}`), and `layout["Umbraco.BlockList"]` (array of `{contentKey,contentUdi,settingsKey,settingsUdi}`). The website seeder builds this JSON directly when creating sample pages.

## Reading content (ModelsBuilder = Nothing)

`Umbraco.CMS.ModelsBuilder.ModelsMode` is set to **`Nothing`** in `appsettings.json`. There are no generated typed models. Read all content untyped:

```csharp
var title = page.Value<string>(WebsiteAliases.HeroTitle);
var media = page.Value<MediaWithCrops>(WebsiteAliases.HeroImage);
```

Do not introduce or assume ModelsBuilder-generated classes.

## Razor is runtime-compiled

`RazorCompileOnBuild` and `RazorCompileOnPublish` are `false` (needed for the backoffice to work). Therefore:

- `dotnet build` does not compile `.cshtml` files, so it will not surface view errors. Validate views by running the app and requesting the page.
- Known trap: naming a `foreach` variable `page` breaks compilation, because `@page.Url()` is parsed as the `@page` Razor directive. Use another name such as `item`.

## Routing and URLs

- Ticketing links use fixed MVC routes (for example `/tickets/event/{sqid}`), where `{sqid}` is a Sqids-encoded id. They do not depend on Umbraco content-node URLs, so reordering root nodes is safe. Public/admin routes are English (`/cart`, `/checkout`, `/seasons`, `/scan`, `/admin/members`, `/admin/season-passes`, `/admin/event-tickets`, `/admin/flex-tickets`); the seasons folder content node is renamed to "Seasons" so it serves at `/seasons`.
- The public homepage is the first `flexPage` at the content root. The website seeder sorts it to the first root position so it serves at `/`, which demotes the ticketing content node's URL harmlessly (ticketing uses the fixed routes above).

## Adaptive navigation

`Views/Shared/_SiteLayout.cshtml` renders a shared nav. When a page opens with a hero or header block, `ViewBag.TransparentNav = true` makes the fixed bar float transparent over the image with white text; a scroll handler switches it to a solid white background with black text past 60px. Pages without a leading hero/header get a solid nav from the start.

## Infrastructure and boot

- **Database**: **Azure SQL in every environment** (dev and prod). `appsettings.json` sets provider `Microsoft.Data.SqlClient` with an empty DSN; the real connection string is injected as the App Service app setting `ConnectionStrings__umbracoDbDSN` (prod) or via user secrets pointing at the shared dev database (local). The SQL Server provider ships transitively with `Umbraco.Cms`. There is no SQLite bootstrap anymore (the earlier SQLite + WAL setup was removed when dev and prod were unified on Azure SQL). `BackgroundServiceExceptionBehavior.Ignore` is set so a transient Azure SQL timeout inside a hosted background service cannot crash the whole app.
- **Deployment**: GitHub Actions (`.github/workflows/deploy.yml`) build/publish/ZipDeploy to an Azure App Service in resource group `RG_RedAnts` on push to `main`. One-time provisioning script and details in `deploy/README.md` and `deploy/azure-setup.sh`.
- **Culture**: default thread culture is `de-CH`, set on `DefaultThreadCurrentCulture`/`UICulture` at the top of `Program.cs`. This is deliberately global, not display-only: Blazor `@bind`, implicit `ToString()`, and number/date parsing all follow `de-CH` (dd.MM.yyyy, apostrophe thousands separator). **Do not reach for `UseRequestLocalization` to localize display strings**: Umbraco's request pipeline overrides the request culture with the content language of the install (English by default), so a negotiated `Accept-Language` culture never survives to view rendering on content pages — that is exactly why weekday names once rendered in English for German visitors. Localized display (currently the weekday on the event page) instead formats explicitly with `DisplayCulture.From(Request)` (`Features/Ticketing/Public/DisplayCulture.cs`), which reads the browser's top `Accept-Language`: English browsers get English day names, everything else gets `de-CH`. Date/number layout stays Swiss everywhere.
- **Development**: OpenIddict transport security is disabled in Development so the backoffice works over HTTP; unattended install/upgrade is enabled.
- **External services**: Payrexx, Brevo, and Turnstile are configured via `appsettings` / user secrets. Secrets are never hardcoded; Development uses Cloudflare test Turnstile keys and empty Payrexx credentials.

## Design rationale and gotchas

The codebase carries no inline comments (the code is meant to speak for itself). The non-obvious "why" behind certain constructs, the kind a future developer would otherwise rediscover the hard way, is recorded here instead.

### Persistence and schema migration
- **Provider-aware ALTER TABLE.** Umbraco's `Alter.Table` / `Delete.Column` builders throw `NotSupportedException` on SQLite, which once aborted the whole migration. The migration therefore detects the provider (`Database.DatabaseType.GetType().Name.Contains("SQLite")`) and runs raw `Execute.Sql("ALTER TABLE ...")` on SQLite while using the Umbraco builder elsewhere. Now that every environment runs on Azure SQL the SQLite branch is dead, but any future additive column change should keep following the same guarded pattern (see `AddFlexTicketBundles`, `AdjustMemberCardColumns`).
- **Member cards have no `Price` column** by design (only season passes are priced); the legacy column was dropped.

### Blazor Server backoffice
- **`UmbracoEventStatusEditor` wraps `IContentService.Save`/`Publish` in `Task.Run` on purpose.** Those APIs are sync-over-async; invoked directly on a Blazor Server circuit's `SynchronizationContext` they deadlock (the UI hangs with no exception). `Task.Run` moves them off that context while still propagating exceptions. Do not "simplify" it away.
- **Umbraco readers wrap reads in `EnsureUmbracoContext()`** because they are also called from the Blazor admin over SignalR, which has no request-scoped Umbraco context. The call is nest-safe, so the MVC path is unaffected.
- **`TicketingAdminState` is per-circuit shared state.** Blazor Server scopes it to the SignalR connection, so every admin tab shares one instance for the session. Contract: on init a tab uses `SelectedSeasonId` when set and still valid, otherwise falls back to the current season and writes that back; on dropdown change it writes back. A fresh circuit (app reload) starts null, so the first tab shown re-establishes the current-season default. It also carries `SelectedEventId`/`SelectedBundleId` and raises `Changed` on any set so `TicketingAdminComponent` can mirror the position into the URL.
- **Admin position lives in the URL query.** `TicketingAdminComponent` parses `tab`/`season`/`event`/`bundle` on init (seeding `TicketingAdminState` before any child mounts) and rewrites them via `NavigateTo(replace: true)` whenever the tab or state changes, so a copied link reopens the same tab, season and event/bundle. URL writes are gated behind the first `OnAfterRender`: the host renders the component with `render-mode="Server"` (no router), so navigating during non-interactive init would throw and there is no `LocationChanged` re-render loop to worry about afterwards. `event`/`bundle` are only emitted on the tabs that own them; the shared state keeps the last value so detouring through another tab and back restores it.
- **Narrow inline write-ports.** `IEventStatusEditor` and `IEventAdmissionQuota` are kept separate from the read-only `CatalogPorts` so the catalog read contract is not widened; setting status republishes the node so the published cache and all catalog readers reflect it. Admin report readers query the Sales tables directly through `IScopeProvider` rather than reusing ticket repositories, to stay decoupled from in-flight repository and port changes.
- **Admin modals close on `@onmousedown`, not click,** so releasing a text selection outside the modal does not dismiss it.
- **Inline editors auto-focus on open and close on blur.** `InlineEditBase` focuses the editor element on the first render after `BeginEdit` (via `EditorElement`), which is what lets a click-away raise `blur` at all; `HandleBlur` then abandons the edit exactly like ESC. The blur handler is guarded against `ConfirmOpen`/`Busy` because opening the confirm dialog (or clicking its buttons) blurs the editor, and an unguarded close would dismiss the pending save. `@onchange` fires before `@onblur`, so a changed value has already raised the confirm dialog by the time blur is handled.

### Ticket QR tokens and scanning
- Tokens are compact HMAC-signed strings (`RA1.<payload>.<sig>`), **signed but not encrypted**: the fields are readable because the scanner must read them, and the signature (a truncated 128-bit HMAC-SHA256) makes them unforgeable without the server secret. `Create` and `TryVerify` must truncate identically for the `FixedTimeEquals` compare. Production must set `Tickets:QrSecret`; dev falls back to a key derived from the Umbraco global id so tokens survive restarts (logged as a warning).
- `ScopeId` is the `EventId` for single-admission tickets and the `SeasonId` for passes and member cards. The QR encodes the **absolute** `/ticket/{token}` URL so a generic phone camera opens the web ticket, while the door scanner takes the last path segment as the token.
- `IQrCodeRenderer` uses QRCoder's managed encoders (no `System.Drawing`) so it runs on the Linux App Service; ECC level Q is a deliberate glare/wear versus size tradeoff.
- Admission model: a first scan checks a person in, a re-scan checks them out (so a ticket becomes free again on exit). Ticketed admissions are never blocked by the hall quota (the quota is informational and only gates anonymous free walk-up entries). A `SeasonSingle` flexticket binds to the first event it is admitted at (`RedeemedEventId`) and only that event admits it thereafter.
- **Helper-to-scan attribution is by name, not id.** Each visit-log row stores `ScannedBy`, a free-text operator name; a logged-in helper writes their `Helper.FullName` (see `HelperNameAsync` in `Program.cs`). The helper admin's per-helper scan overview therefore matches `HelperScanRow.Person == Helper.FullName` rather than a foreign key, and check-ins (`Type = 0`) versus check-outs (`Type = 1`) are summed separately. Two helpers with an identical full name would merge, and a manually typed operator name that happens to equal a helper's name would be attributed to that helper; both are accepted tradeoffs of the name-based logging.
- **Helper login passwords are pre-declined adjective+noun word lists, not runtime-inflected.** `HelperPassword` holds six literal arrays: nouns split into `Masculine`/`Feminine`/`Neuter` and the adjective in all three genders (`-er`/`-e`/`-es`, indeclinable ones like `Lila`/`Rosa` unchanged). `Generate` picks a noun and takes the matching-gender adjective form, so every password is grammatically correct by construction with no declension logic to get wrong. The lists were curated (nice, concrete, everyday words), the gender taken from an authoritative German noun dataset rather than guessed, umlauts transliterated to `ae/oe/ue/ss` so passwords stay ASCII-typable, and words filtered out that are sensitive/insulting, very long, or unusual. To extend or regenerate, edit the arrays directly.

### Content seeding
- **`EnsureLabel` matches the data type by Name, not `EditorAlias`.** Several built-in `Umbraco.Label` types exist; an alias match grabs "Label (pixels)" and renders a string link as "px". The Name lookup plus explicit `umbracoDataValueType = "STRING"` avoids that.
- The website seeder runs after the ticketing seeder (`[ComposeAfter]`) so the sample homepage's event-list block can reference an existing season. Template aliases are prefixed (e.g. `TicketEvent`) so they do not collide with the MVC purchase views through Umbraco's root view-location expander.
- Access-link fields are computed from the published Umbraco URL (empty until first publish, filled on the next save); writes happen only on an actual change to avoid an infinite `ContentSaved` re-trigger loop.
- `EventAdmissionReportReader` counts asymmetrically on purpose: card visits use `COUNT(DISTINCT TicketUuid)`, while FreeEntry admissions carry no ticket uuid and use `COUNT(*)`. Do not unify these.
- **The venue picker's `startNodeId` config is a bare document Key (GUID), not a UDI.** In the new backoffice the Document Picker UI (`Umb.PropertyEditorUi.DocumentPicker`) reads the `startNodeId` config value and passes it straight through as the tree root's `unique`, which is a plain GUID. Storing the folder UDI (`umb://document/...`) there resolves to no node, so the venue tree comes up empty and the event's "Ort" appears unchangeable. `EnsureVenuePicker` therefore writes `venuesFolder.Key.ToString()`; the idempotent update path migrates any old UDI value on the next boot. The server-side `ContentPickerConfiguration` only exposes `ignoreUserStartNodes` in v17, so the start-node restriction is a purely client-config concern.

### Purchase, orders, exports
- **Checkout is a pseudo-payment:** no gateway is called, every sale succeeds and is persisted as paid, and one `EventTicket` (its own Uuid and online ticket) is issued per unit. VAT is 0 by design because Swiss sports entry fees are exempt (Art. 21 Abs. 2 Ziff. 15 MWSTG); the model stays VAT-ready for the day the club becomes liable.
- `CartController.Add` always resolves price and name server-side and never trusts a posted price, and distinguishes a fetch call from a full post via the `X-Requested-With: fetch` header (progressive enhancement: without JS the same form posts normally and the server redirects).
- **`PaymentMethod` name clash:** the type exists in both `Domain.Ticketing` and `Domain.Ticketing.Sales`. The Sales one is authoritative and kept as an aliased `using` in `OrderRepository`; removing or reordering that alias reintroduces an ambiguous-reference compile error.
- **Season add-on scope (`SeasonAddOn.Scope`):** each add-on is either `PerPass` (charged once per season pass, so its cost multiplies with the pass line quantity, e.g. a physical card, opt-in and off by default) or `PerOrder` (charged once per whole order regardless of how many passes, e.g. a Livestream subscription). `PerOrder` add-ons live in a separate `Cart.OrderAddOns` collection (not on the pass line) and are deduplicated by id, which is what enforces "max one per order" even though the public season page still offers the checkbox on every category card. In the fulfillment snapshot both scopes flow through `FulfillmentSnapshot.AddOns`, but `PerOrder` entries are written with quantity 1 so `FulfillAsync` (webhook-deferred) stays scope-agnostic. The additive `Scope` column defaults to 0 (`PerPass`), so add-ons that existed before this change keep the old per-pass behaviour until an admin sets the scope.
- **Serienbrief export (`FlexBundleExportController`):** RTF keeps this dependency-free (Word opens it natively). Each QR is embedded as a PNG in an RTF `\pict\pngblip` group; the intrinsic pixel size is read from the PNG IHDR (big-endian width/height at bytes 16 to 19 / 20 to 23), converted to himetric (0.01 mm) at 96 dpi for `picw`/`pich`, while `picwgoal`/`pichgoal = 2551` twips renders about 4.5 cm square. CSV exports prepend a UTF-8 BOM so Excel picks the right encoding; `MemberCsv` defaults to `;` (Swiss Excel) and also writes a BOM for umlauts.

### Live-test security gate
- **`MemberExportController` returns valid admission tokens plus member PII** and is currently protected only by the global HTTP Basic gate. Proper `[Authorize(BackOfficeAuthenticationType)]` must be added before the site goes public (the other admin controllers already have it).
- The Basic-auth gate (active only when `BasicAuth:Username`/`Password` are configured) is a soft live-test lockdown, not real security. It lets the backoffice-internal paths through so Umbraco's own auth governs them (`/umbraco`, the admin dashboard host `/admin/ticketing`, and the Blazor Server hub and framework scripts `/_blazor` and `/_framework`): the backoffice authenticates with a Bearer token that would never match a Basic header, so gating those would 401-loop the login or make the dashboard iframe re-prompt. `/admin/members/*` stays gated on purpose, because its export controller carries no authorization of its own, so the Basic gate is its only guard. After a successful challenge the gate sets a SHA-256 unlock cookie (cached Basic credentials do not ride along on the Blazor SignalR WebSocket handshake or background fetches, but cookies do), accepts both UTF-8 and Latin-1 decodings of the credentials (so umlaut passwords match), and sends `WWW-Authenticate` only on top-level navigations (via `Sec-Fetch-Mode: navigate`) so background requests fail silently instead of re-prompting the password dialog. Host-based landing: the `scan.` subdomain root redirects to `/scan`, while the `tickets.` subdomain root is left as the public event list.

### Front-end
- **One ticket look across four surfaces (online ticket, checkout confirmation, e-mail, PDF).** The design is a credit-card (ISO ID-1) portrait card, monochrome Red Ants red (`#D02D38`, `#B0242E` for the small red date); the ticket type is read from the kicker + title only (`TicketDisplay.Kicker`/`TypeLabel`), no per-type colour or emblem. The two Razor surfaces share `Views/Partials/_TicketCard.cshtml` + `wwwroot/css/ticket-card.css` (`rtk-*`, driven by `TicketCardModel`); the card height is **content-driven on screen** (a fixed `aspect-ratio` + `overflow:hidden` clips on mobile) and only forced to exact 54×85.6 mm in `@media print`. The e-mail rebuilds the same card as inline-styled tables (no SVG/notches) and the PDF rebuilds it in QuestPDF on a real 54×85.6 mm page. The abriss notches use a `--rtk-notch` variable set to the surrounding background per host. `ConfirmationTicket` carries `Type` + `DateText` (resolved in `CheckoutController.FulfillAsync` via `IEvents`/`ISeasons`) so the confirmation card shows the same fields as the others.
- **The custom 404 page is served by a response-buffering middleware, not `UseStatusCodePagesWithReExecute`.** Umbraco writes its own "nonodes" body for unmatched content routes, which starts the response and makes the re-execute middleware a no-op. The middleware in `Program.cs` therefore swaps `Response.Body` for a `MemoryStream` on extension-less page GETs (skipping `/umbraco`, `/App_Plugins`, `/_blazor`, `/_framework`, `/api`, `/ticket`, `/warmup`); if the final status is 404 it discards the buffered body and writes the static `wwwroot/404.html` (branded, fully static, so no view engine or content node is needed), otherwise it copies the buffer through unchanged.
- **Security headers are set in one middleware via `Response.OnStarting`** (so they apply to redirects and short-circuits too). `X-Content-Type-Options`, `Referrer-Policy` and (over HTTPS) `Strict-Transport-Security` are global; the `Server` header is removed via Kestrel `AddServerHeader=false`. `X-Frame-Options: SAMEORIGIN` is set everywhere except `/next/embed` (which needs cross-site framing and sets its own `frame-ancestors *`). A conservative `Content-Security-Policy` is applied only to public front-end responses and skips the backoffice, Blazor, scanner and embed paths, because those need inline/eval and their own framing rules; the policy keeps `'unsafe-inline'` for scripts and styles because the public views rely on inline styles and handlers throughout.
- **`RuntimeMode` is `Production` in base config, overridden to `BackofficeDevelopment` in `appsettings.Development.json`.** Umbraco's production-mode validators require `Global:UseHttps=true` (set in base, `false` in dev), `WebRouting:UmbracoApplicationUrl` (set), `ModelsBuilder:ModelsMode=Nothing` (set) and a Release build (CI publishes Release). Runtime Razor compilation stays enabled: the `RazorRuntimeCompilationValidator` only fails for `ModelsMode=InMemoryAuto`, which is not used here.
- **`/warmup` server-side preloads the core runtime-compiled views** (`/`, `/ticketing/`, `/seasons/`, `/next`, `/next/embed`, `/umbraco`, and the first public season detail) by looping HTTP requests back to its own host, carrying the site-gate cookie when the gate is active so the real views compile rather than the gate page. The deploy warmup step calls `/warmup` after the app answers.
- **`/next/embed` is a standalone slim widget for cross-site iframe embedding** (the club's main page). It renders `NextEventEmbed.cshtml` with `Layout = null` (no site nav/footer, self-contained styles) showing the next public event's title, home-vs-opponent logos and a "Hier Tickets kaufen" link that navigates the top window (`target="_top"`) to the event's absolute purchase URL. The action sends `Content-Security-Policy: frame-ancestors *` so it can be framed anywhere; this is deliberate and safe because the widget is public, read-only and carries no forms or session data. `/next` (without `/embed`) still redirects to that same next event.
- **The scanner service worker (`scanner-sw.js`) exists solely to make the app installable** (browsers require a registered service worker with a fetch handler); it does no offline caching because the scanner needs a live Blazor Server circuit, so its `fetch` handler is a deliberate network passthrough.
- `wwwroot/css/ticketing-admin.css` is the single source of truth for all five admin tabs (identical row/header heights, dropdown width via `--ta-field-w`, toolbar, sortable headers, edit-icon column, overlay). Keeping every rule there rather than in per-component `<style>` blocks is what keeps the tabs visually identical.
- **Editable-value convention:** an inline-editable cell renders the `InlineEditBase` `.ta-inline` button, which now carries a persistent dashed underline (muted, turns accent on hover); a plain fixed cell has none. This is the shared visual cue "this value can be clicked to edit" — reuse `.ta-inline` (via the `Inline*Edit` components) for editable values and leave read-only values as plain text so the two are always told apart.
- **Season tiers and their promotions (Sonderaktion):** a promotion is a second `PriceTier` whose `PromoOfTierId` points at the base tier (its `SeasonCategoryPrice` row carries the action price/quota/until). The season pricing modal shows each promo as a tinted sub-row under its tier, toggled per tier with the `.ta-promo-toggle` pill, so any tier — including custom ones — can get an action. On save, `SaveEdit` renumbers the kept tiers' `SortOrder` to their list index before persisting, so the edited rows and the re-fetched tiers (both ordered by `SortOrder`) pair up positionally without ambiguity even for freshly added tiers.
- The web ticket gives each ticket type its own header accent colour (`wt-head--{spiel|flex|saison|member|free}`) so a stack of tickets is told apart at a glance; the type name still appears as text, so colour is a redundant cue, not the sole signal.
- **Ticket delivery (PDF):** every ticket is downloadable at `/ticket/{token}/pdf` (QuestPDF, Community licence set once at startup in `TicketDeliveryComposer`; the QR is the same signed ticket-page URL). QuestPDF relies on `SkiaSharp.NativeAssets.Linux.NoDependencies` and its embedded default font, so it renders on Azure App Service Linux without extra system packages.
- Team logos (home team / opponent) render inside a fixed `.team-logo` box (`wwwroot/css/site.css`, `object-fit: contain`) shared by the ticketing home, the season page, the event page and the `eventListElement` block. Without a fixed width, a wide/panoramic club logo at a fixed height stretched the row and distorted the layout, so the box constrains both dimensions and letterboxes each logo to a uniform size.
- The event page embeds the venue map and the "Route öffnen" directions link from the venue's `googleGeoId`, which is treated as a Google **Place ID**: the classic keyless embed uses `q=place_id:<id>` and the directions link uses the official Maps URL scheme `?api=1&destination=<name>&destination_place_id=<id>`. When a venue has no geo id, both fall back to the `"<name>, Winterthur"` text query so a map still renders.
- Footer legal links (Impressum, Datenschutzerklärung, AGB) are content-driven: they are `legalPage` nodes grouped under a single `footerFolder` root node ("Footer"), and `_SiteLayout` renders one link per published child ordered by sort order, so adding a node under Footer adds a footer link with no code change. `LegalPageContentFinder` resolves the clean single-segment URL (`/impressum`, `/agb`, …) by matching the node's `slug` regardless of where the node sits, so the pages keep root-level URLs even though they live under the folder. The startup seeder creates the folder, moves any pre-existing root-level legal pages under it and republishes them (Umbraco's `Move` drops the published state), then seeds an AGB placeholder. Scanner and Verwaltung stay hard-coded in a separate, de-emphasised footer row because they are internal routes, not content. Caveat: while the pre-launch gate is active, a new footer slug must also be added to `IsExempt` in `Program.cs` or the gate blocks it.
